# π₀.₅ (Pi05) Policy

π₀.₅ is a **Vision-Language-Action model with open-world generalization**, from Physical Intelligence. The LeRobot implementation is adapted from their open source [OpenPI](https://github.com/Physical-Intelligence/openpi) repository.

## Model Overview

π₀.₅ represents a significant evolution from π₀, developed by [Physical Intelligence](https://www.physicalintelligence.company/blog/pi05) to address a big challenge in robotics: **open-world generalization**. While robots can perform impressive tasks in controlled environments, π₀.₅ is designed to generalize to entirely new environments and situations that were never seen during training.

### The Generalization Challenge

As Physical Intelligence explains, the fundamental challenge isn't performing tasks of agility or dexterity, but generalization, the ability to correctly perform tasks in new settings with new objects. Consider a robot cleaning different homes: each home has different objects in different places. Generalization must occur at multiple levels:

- **Physical Level**: Understanding how to pick up a spoon (by the handle) or plate (by the edge), even with unseen objects in cluttered environments
- **Semantic Level**: Understanding task semantics, where to put clothes and shoes (laundry hamper, not on the bed), and what tools are appropriate for cleaning spills
- **Environmental Level**: Adapting to "messy" real-world environments like homes, grocery stores, offices, and hospitals

### Co-Training on Heterogeneous Data

The breakthrough innovation in π₀.₅ is **co-training on heterogeneous data sources**. The model learns from:

1. **Multimodal Web Data**: Image captioning, visual question answering, object detection
2. **Verbal Instructions**: Humans coaching robots through complex tasks step-by-step
3. **Subtask Commands**: High-level semantic behavior labels (e.g., "pick up the pillow" for an unmade bed)
4. **Cross-Embodiment Robot Data**: Data from various robot platforms with different capabilities
5. **Multi-Environment Data**: Static robots deployed across many different homes
6. **Mobile Manipulation Data**: ~400 hours of mobile robot demonstrations

This diverse training mixture creates a "curriculum" that enables generalization across physical, visual, and semantic levels simultaneously.

To use π₀.₅ in your LeRobot configuration, specify the policy type as:

```python
policy.type=pi05
```

## Training

### Training Command Example

Here's a complete training command for finetuning the base π₀.₅ model on your own dataset:

```bash
lerobot-train \
    --dataset.repo_id=your_dataset \
    --policy.type=pi05 \
    --output_dir=./outputs/pi05_training \
    --job_name=pi05_training \
    --policy.repo_id=your_repo_id \
    --policy.pretrained_path=lerobot/pi05_base \
    --policy.compile_model=true \
    --policy.gradient_checkpointing=true \
    --wandb.enable=true \
    --policy.dtype=bfloat16 \
    --policy.freeze_vision_encoder=false \
    --policy.train_expert_only=false \
    --steps=3000 \
    --policy.device=cuda \
    --batch_size=32
```

### Key Training Parameters

- **`--policy.compile_model=true`**: Enables model compilation for faster training
- **`--policy.gradient_checkpointing=true`**: Reduces memory usage significantly during training
- **`--policy.dtype=bfloat16`**: Use mixed precision training for efficiency
- **`--batch_size=32`**: Batch size for training, adapt this based on your GPU memory
- **`--policy.pretrained_path=lerobot/pi05_base`**: The base π₀.₅ model you want to finetune, options are:
  - [lerobot/pi05_base](https://huggingface.co/lerobot/pi05_base)
  - [lerobot/pi05_libero](https://huggingface.co/lerobot/pi05_libero) (specifically trained on the Libero dataset)

### Training Parameters Explained

Fresh π₀.₅ configurations use the stable LeRobot AdamW baseline of `2.5e-5`. After warmup, the
learning rate follows a cosine decay to `2.5e-6`. Loading or resuming a checkpoint preserves the
optimizer values saved in that checkpoint.

Gradient clipping is policy-specific. The action-side gradient RMS is capped at 10 times the VLM
gradient RMS, and only action-side gradients are rescaled when that cap is exceeded. VLM gradients
are left unchanged by that policy-specific constraint. A global norm cap of `1.0` is applied first
as a model-wide safety bound.

PI0.5 also provides experimental counterfactual action-space optimizer control (CABO). CABO simulates
the next VLM and action-side AdamW updates and estimates their linearized effects on predicted flow
velocity. Its default `residual` mode treats the action side as the primary learner: the action update
always runs at full scale, while the VLM update runs only when it is predicted to reduce the
flow-matching residual left after that action update. Action-gradient clipping remains enabled in this
mode:

```bash
lerobot-train \
    ... \
    --policy.cabo_enabled=true \
    --policy.cabo_control_mode=residual \
    --policy.cabo_residual_regularization=0.1 \
    --policy.cabo_residual_vlm_min_scale=0.1 \
    --policy.cabo_residual_vlm_max_scale=1.0 \
    --policy.cabo_vlm_optimizer_lr=2.5e-5 \
    --policy.cabo_action_optimizer_lr=1e-4 \
    --policy.cabo_warmup_steps=1000 \
    --policy.cabo_probe_interval=4 \
    --policy.cabo_probe_batch_size=4 \
    --policy.cabo_num_projections=4
```

Let `e = predicted_velocity - target_velocity`, and let `dv` and `da` be the linearized velocity
changes from the full VLM and action-side candidate updates. Residual mode chooses a VLM scale `s`
that minimizes

`mean(||e + da + s * dv||^2) + regularization * s^2 * mean(||dv||^2)`.

Its closed-form scale is clipped to
`[cabo_residual_vlm_min_scale, cabo_residual_vlm_max_scale]`. The default floor is zero and therefore
preserves the exact controller. For noisy, heterogeneous datasets, a small positive floor prevents
one minibatch from completely freezing representation learning. `cabo_residual_regularization` is
dimensionless and discourages using VLM capacity for small, noisy predicted gains.

`cabo_vlm_optimizer_lr` and `cabo_action_optimizer_lr` optionally set different base learning rates
for the two named AdamW groups. This is useful when the action expert needs to adapt faster without
subjecting the pretrained VLM to the same aggressive rate. `cabo_warmup_steps` keeps both groups at
full scale and defers CABO probes while the optimizer moments stabilize.

The earlier `budget` and `balance` modes remain available for ablations and checkpoint compatibility.
`budget` keeps the VLM at full scale and limits action-side drift. `balance` ignores task-residual
direction and makes the two marginal drift magnitudes similar using reciprocal scales. These legacy
modes disable the action-gradient clipping hook:

```bash
lerobot-train \
    ... \
    --policy.cabo_enabled=true \
    --policy.cabo_control_mode=balance \
    --policy.cabo_balance_max_scale=2.0 \
    --policy.cabo_probe_interval=1 \
    --policy.cabo_probe_batch_size=4 \
    --policy.cabo_num_projections=16 \
    --policy.cabo_drift_ema_decay=0.0
```

CABO estimates the VLM drift `Dv`, full action drift `Da`, cross drift `Cva`, residual energy, and the
residual alignment of both candidate updates with shared Rademacher projections. Padded action
timesteps are masked out of the probe and its normalization; partially padded trajectories still
contribute their valid timesteps. Probe rows are sampled through a deterministic private RNG, so
enabling CABO does not consume the training RNG stream.

The controller reuses its latest scale or scales between probes. Its guarantee is therefore relative,
sampled, and first-order rather than a bound on the exact post-update network or final denoised action
chunk. With bfloat16 parameters, CABO simulates native-dtype AdamW moments and the full-scale
representable parameter write, but scaling that quantized update is still treated as linear.

CABO currently supports single-device and DDP training, but not FSDP. In DDP, every rank participates
in each scheduled probe collective, and statistics are weighted by the number of valid action output
elements on that rank. It requires both the VLM and action side to be trainable, so it cannot be
combined with `train_expert_only=true`. CABO also requires the policy training preset and its two
named AdamW parameter groups. Enable it in a new run initialized from model weights; an older
single-group optimizer checkpoint cannot be resumed directly with CABO enabled. A non-finite training
gradient or probe causes all ranks to skip the complete optimizer and scheduler step without advancing
the CABO controller state.

| Parameter                       | Default  | Description                                                |
| ------------------------------- | -------- | ---------------------------------------------------------- |
| `optimizer_lr`                  | `2.5e-5` | Stable AdamW baseline learning rate                         |
| `scheduler_decay_lr`            | `2.5e-6` | Learning-rate floor after cosine decay                      |
| `clip_action_head_by_vlm`       | `true`   | Enable the action-side RMS cap                              |
| `action_head_grad_clip_ratio`   | `10.0`   | Enforce `action_rms <= 10 * vlm_rms`                        |
| `optimizer_grad_clip_norm`      | `1.0`    | Model-wide gradient norm safety cap                         |
| `cabo_enabled`                  | `false`  | Enable counterfactual action-space update control                  |
| `cabo_control_mode`             | `residual` | Use action-first VLM residual compensation; legacy: `budget`, `balance` |
| `cabo_residual_regularization`  | `0.1`    | Penalize VLM drift in the residual-mode one-step objective            |
| `cabo_residual_vlm_min_scale`   | `0.0`    | Optional VLM learning floor for noisy probes                           |
| `cabo_residual_vlm_max_scale`   | `1.0`    | Maximum VLM learning-rate multiplier in residual mode                  |
| `cabo_vlm_optimizer_lr`         | `None`   | Optional CABO VLM-group base LR; inherits `optimizer_lr`               |
| `cabo_action_optimizer_lr`      | `None`   | Optional CABO action-group base LR; inherits `optimizer_lr`            |
| `cabo_warmup_steps`             | `0`      | Steps before CABO begins probing                                      |
| `cabo_balance_max_scale`        | `2.0`    | Maximum amplification in balance mode; minimum scale is its reciprocal     |
| `cabo_action_drift_ratio`       | `0.1`    | Relative linearized allowance over VLM-only RMS drift              |
| `cabo_probe_interval`           | `8`      | Steps between CABO flow-velocity probes                            |
| `cabo_probe_batch_size`         | `1`      | Per-rank rows with valid timesteps used by each CABO probe          |
| `cabo_num_projections`          | `4`      | Shared random projections used to estimate drift and its cross term |
| `cabo_base_action_scale`        | `0.1`    | Action-only allowance that relaxes the pure relative bound          |
| `cabo_negative_cross_discount`  | `0.5`    | Fraction of measured cancellation credit trusted by the controller |
| `freeze_vision_encoder`         | `false`  | Do not freeze the vision encoder                            |
| `train_expert_only`             | `false`  | Do not freeze the VLM; train all parameters                 |

**💡 Tip**: Setting `train_expert_only=true` freezes the VLM and trains only the action expert and projections, allowing finetuning with reduced memory usage.

If your dataset is not converted with `quantiles`, you can convert it with the following command:

```bash
python src/lerobot/scripts/augment_dataset_quantile_stats.py \
    --repo-id=your_dataset \
```

Or train pi05 with this normalization mapping: `--policy.normalization_mapping='{"ACTION": "MEAN_STD", "STATE": "MEAN_STD", "VISUAL": "IDENTITY"}'`

## Relative Actions

By default, π₀.₅ predicts absolute actions. You can enable **relative actions** so the model predicts offsets relative to the current robot state. This can improve training stability for certain setups.

To use relative actions, first recompute your dataset stats in relative space via the CLI:

```bash
lerobot-edit-dataset \
    --repo_id your_dataset \
    --operation.type recompute_stats \
    --operation.relative_action true \
    --operation.chunk_size 50 \
    --operation.relative_exclude_joints "['gripper']" \
    --push_to_hub true
```

Or equivalently in Python:

```python
from lerobot.datasets import LeRobotDataset, recompute_stats

dataset = LeRobotDataset("your_dataset")
recompute_stats(dataset, relative_action=True, chunk_size=50, relative_exclude_joints=["gripper"])
dataset.push_to_hub()
```

The `chunk_size` should match your policy's `chunk_size` (default 50 for π₀.₅). `relative_exclude_joints` lists joint names that should remain in absolute space (e.g. gripper commands). Use `--push_to_hub true` to upload the updated stats to the Hub.

Then train with relative actions enabled:

```bash
lerobot-train \
    --dataset.repo_id=your_dataset \
    --policy.type=pi05 \
    --policy.use_relative_actions=true \
    --policy.relative_exclude_joints='["gripper"]' \
    ...
```

## Performance Results

### Libero Benchmark Results

π₀.₅ has demonstrated strong performance on the Libero benchmark suite. To compare and test its LeRobot implementation, we finetuned the libero base model for an additional 6k steps on the Libero dataset and compared the results to the OpenPI reference results.

| Benchmark          | LeRobot Implementation | OpenPI Reference |
| ------------------ | ---------------------- | ---------------- |
| **Libero Spatial** | 97.0%                  | 98.8%            |
| **Libero Object**  | 99.0%                  | 98.2%            |
| **Libero Goal**    | 98.0%                  | 98.0%            |
| **Libero 10**      | 96.0%                  | 92.4%            |
| **Average**        | 97.5%                  | 96.85%           |

These results demonstrate π₀.₅'s strong generalization capabilities across diverse robotic manipulation tasks. To reproduce these results, you can follow the instructions in the [Libero](https://huggingface.co/docs/lerobot/libero) section.
